# Design High Availability Architecture - Complete Notes

## Overview

High Availability (HA) Architecture ensures systems remain operational with minimal downtime. This is a critical system design topic asked in various forms.

---

## Different Ways This Question is Asked

| Question Variant | Core Focus |
|------------------|------------|
| Design High Availability Architecture | Overall system resilience |
| Design Data Resilience Architecture | Recovery from failures |
| Design for 99.999% Availability | Five 9s uptime guarantee |
| Avoid Single Point of Failure (SPOF) | Redundancy at every layer |
| Active-Passive vs Active-Active | Architecture pattern comparison |

---

## Understanding Availability Percentages

| Availability | Downtime/Year | Downtime/Month | Downtime/Day |
|--------------|---------------|----------------|--------------|
| 99% (Two 9s) | 3.65 days | 7.3 hours | 14.4 minutes |
| 99.9% (Three 9s) | 8.76 hours | 43.8 minutes | 1.44 minutes |
| 99.99% (Four 9s) | 52.56 minutes | 4.38 minutes | 8.64 seconds |
| **99.999% (Five 9s)** | **5.26 minutes** | **26.3 seconds** | **0.86 seconds** |

**Goal: Every company aims for Five 9s (99.999%)**

## Single Node Architecture (The Problem)

### Architecture

```
┌──────────┐      ┌───────────────┐      ┌─────────────────────┐      ┌──────────────┐
│          │      │               │      │  Application Layer  │      │              │
│  Client  │─────►│ Load Balancer │─────►│  ┌─────┐ ┌─────┐   │─────►│  Primary DB  │
│          │      │               │      │  │App X│ │App Y│   │      │   (Single)   │
└──────────┘      └───────────────┘      │  └─────┘ └─────┘   │      └──────────────┘
                                         │  ┌─────┐           │
                                         │  │App Z│           │
                                         │  └─────┘           │
                                         └─────────────────────┘
```

### What Happens When DB Fails?

```
Client ──► Load Balancer ──► Application ──► DB [DOWN] ✗
                                                │
                                                ▼
                                    ┌─────────────────────┐
                                    │   TOTAL FAILURE     │
                                    │   • All Reads FAIL  │
                                    │   • All Writes FAIL │
                                    │   • System DOWN     │
                                    └─────────────────────┘
```

### Problems with Single Node

| Criteria | Status | Problem |
|----------|--------|---------|
| 99.999% Availability | ❌ NO | System down when DB fails |
| Single Point of Failure | ❌ YES | DB is SPOF |
| Data Resilience | ❌ NO | No automatic recovery |
| Recovery Time | ❌ SLOW | Hours or days to fix |

**Solution: Multi-Node Architecture**

## Multi-Node Architecture Types

```
Multi-Node Architecture
        │
        ├── Active-Passive (Primary-Replica)
        │   └── One Primary, others are backups
        │
        └── Active-Active (Multi-Master)
            └── All nodes are Primary
```

## Active-Passive Architecture

### Also Known As
- Primary-Replica
- Master-Slave
- Read-Write & Read-Only
- Live DB & DR (Disaster Recovery)

### Architecture Diagram

```
                                    DATA CENTER 1 (Mumbai)
                                    ═══════════════════════
                                         [PRIMARY]
┌──────────┐                       ┌─────────────────────────┐
│          │      ┌──────────┐     │   ┌─────────────────┐   │
│  Client  │─────►│   Load   │─────┼──►│ Application     │   │
│          │      │ Balancer │     │   │ Layer           │   │
└──────────┘      └──────────┘     │   └────────┬────────┘   │
                       │           │            │            │
                       │           │            ▼            │
                       │           │   ┌─────────────────┐   │
                       │           │   │   PRIMARY DB    │   │
                       │           │   │  (Read/Write)   │   │
                       │           │   │     [LIVE]      │   │
                       │           │   └────────┬────────┘   │
                       │           └────────────┼────────────┘
                       │                        │
                       │              ONE-WAY SYNC ▼
                       │                        │
                       │           ┌────────────┼────────────┐
                       │           │            ▼            │
                       │           │   ┌─────────────────┐   │
                       │           │   │   REPLICA DB    │   │
                       │           │   │  (Read Only)    │   │
                       └───────────┼──►└─────────────────┘   │
                                   │   ┌─────────────────┐   │
                                   │   │ Application     │   │
                                   │   │ Layer           │   │
                                   │   └─────────────────┘   │
                                   └─────────────────────────┘
                                    DATA CENTER 2 (Pune)
                                    ═══════════════════════
                                    [DISASTER RECOVERY - DR]
```

### Key Characteristics

| Aspect | Description |
|--------|-------------|
| Primary DB | Only ONE can be Primary/Live |
| Replica DB | Read-only, receives sync from Primary |
| Sync Direction | One-way (Primary → Replica) |
| Write Operations | ONLY go to Primary DB |
| Read Operations | Can go to Primary OR Replica |

### Request Flow Scenarios

#### Scenario 1: Request to Primary Data Center (DC1 - Mumbai)

```
WRITE Request:
Client → LB → DC1 App → Primary DB (Mumbai) ✓
                        └── Writes directly

READ Request:
Client → LB → DC1 App → Primary DB (Mumbai) ✓
                        └── Reads directly
```

#### Scenario 2: Request to DR Data Center (DC2 - Pune)

```
WRITE Request:
Client → LB → DC2 App ──────────────────► Primary DB (Mumbai)
                       Cross-region call!
                       (Latency added)

READ Request (Optimized):
Client → LB → DC2 App → Replica DB (Pune) ✓
                        └── Read locally from replica
```

### Why Can't We Write to Replica?

```
┌─────────────────────────────────────────────────────────────┐
│                                                             │
│   Oracle, MySQL, PostgreSQL = NOT Multi-Master              │
│                                                             │
│   • Can only write to ONE designated Primary/Live DB        │
│   • Replica DBs are READ-ONLY                               │
│   • All writes MUST go to Primary                           │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### Failover Process

```
NORMAL OPERATION:
┌─────────────────┐                    ┌─────────────────┐
│    DC1 Mumbai   │      Sync          │    DC2 Pune     │
│   [PRIMARY]     │ ─────────────────► │   [REPLICA]     │
│   Read/Write    │                    │   Read-Only     │
└─────────────────┘                    └─────────────────┘


PRIMARY FAILS! ❌
┌─────────────────┐                    ┌─────────────────┐
│    DC1 Mumbai   │                    │    DC2 Pune     │
│     [DOWN]      │        ?           │   [REPLICA]     │
│       ✗         │                    │   Read-Only     │
└─────────────────┘                    └─────────────────┘


FAILOVER EXECUTED:
┌─────────────────┐                    ┌─────────────────┐
│    DC1 Mumbai   │                    │    DC2 Pune     │
│     [DOWN]      │                    │ [NEW PRIMARY]   │
│       ✗         │                    │   Read/Write    │
└─────────────────┘                    └─────────────────┘
        │
        │ When DC1 recovers...
        ▼
┌─────────────────┐      Sync          ┌─────────────────┐
│    DC1 Mumbai   │ ◄───────────────── │    DC2 Pune     │
│  [NEW REPLICA]  │                    │   [PRIMARY]     │
│   Read-Only     │                    │   Read/Write    │
└─────────────────┘                    └─────────────────┘
```

### Latency Comparison

```
Request to DC1 (Primary):
┌────────────────────────────────────┐
│ Client → LB → App → DB (local)     │
│ Latency: ~1 second                 │
└────────────────────────────────────┘

Write Request to DC2 (DR):
┌────────────────────────────────────────────────────┐
│ Client → LB → DC2 App → Primary DB (Mumbai)        │
│                         └── Cross-region hop       │
│ Latency: ~2 seconds (added network latency)        │
└────────────────────────────────────────────────────┘
```

### Advantages & Disadvantages

| Advantages | Disadvantages |
|------------|---------------|
| ✅ Simple to implement | ❌ Resource underutilization |
| ✅ Works with SQL databases | ❌ Latency for DR write requests |
| ✅ Clear data consistency | ❌ Failover gap (10-15 min) |
| ✅ Easy to understand | ❌ Write bottleneck on Primary |
| ✅ Lower complexity | ❌ Not ideal for write-heavy apps |

## Active-Active Architecture

### Also Known As
- Multi-Master
- Dual-Active
- Multi-Primary

### Database Support

| Database | Multi-Master Support |
|----------|---------------------|
| Oracle | ❌ NO |
| MySQL | ❌ NO |
| PostgreSQL | ❌ NO |
| **Cassandra** | ✅ YES |
| **CockroachDB** | ✅ YES |
| **MongoDB** | ✅ YES (with config) |
| **Most NoSQL** | ✅ YES |

### Architecture Diagram

```
                                    DATA CENTER 1 (Mumbai)
                                    ═══════════════════════
                                      [PRIMARY - ACTIVE]
┌──────────┐                       ┌─────────────────────────┐
│          │      ┌──────────┐     │   ┌─────────────────┐   │
│  Client  │─────►│   Load   │─────┼──►│ Application     │   │
│          │      │ Balancer │     │   │ Layer           │   │
└──────────┘      └──────────┘     │   └────────┬────────┘   │
                       │           │            │            │
                       │           │            ▼            │
                       │           │   ┌─────────────────┐   │
                       │           │   │  PRIMARY DB 1   │   │
                       │           │   │  (Read/Write)   │   │
                       │           │   │     [LIVE]      │   │
                       │           │   └────────┬────────┘   │
                       │           └────────────┼────────────┘
                       │                        │
                       │            BI-DIRECTIONAL SYNC
                       │                   ▲    ▼
                       │                        │
                       │           ┌────────────┼────────────┐
                       │           │            ▼            │
                       │           │   ┌─────────────────┐   │
                       │           │   │  PRIMARY DB 2   │   │
                       │           │   │  (Read/Write)   │   │
                       │           │   │     [LIVE]      │   │
                       └───────────┼──►└─────────────────┘   │
                                   │   ┌─────────────────┐   │
                                   │   │ Application     │   │
                                   │   │ Layer           │   │
                                   │   └─────────────────┘   │
                                   └─────────────────────────┘
                                    DATA CENTER 2 (Pune)
                                    ═══════════════════════
                                      [PRIMARY - ACTIVE]
```

### Key Characteristics

| Aspect | Description |
|--------|-------------|
| Both DBs | Can be Primary/Live |
| Sync Direction | Bi-directional (both ways) |
| Write Operations | Can go to ANY DB |
| Read Operations | Can go to ANY DB |
| Resource Usage | Fully utilized |

### Request Flow

```
ANY Request to DC1 (Mumbai):
Client → LB → DC1 App → DC1 DB ✓
                        └── Served locally (fast!)

ANY Request to DC2 (Pune):
Client → LB → DC2 App → DC2 DB ✓
                        └── Served locally (fast!)

Both DCs can handle both READ and WRITE!
Changes sync bidirectionally in background.
```

### The Complexity: Bi-directional Sync

```
┌──────────────────────────────────────────────────────────────────┐
│                     SYNC CHALLENGES                               │
├──────────────────────────────────────────────────────────────────┤
│                                                                   │
│  CHALLENGE 1: Write Conflict                                      │
│  ┌───────────────┐                     ┌───────────────┐         │
│  │     DC1       │                     │     DC2       │         │
│  │ UPDATE user   │                     │ UPDATE user   │         │
│  │ SET name='A'  │  ◄── Same time! ──► │ SET name='B'  │         │
│  │ WHERE id=1    │      Same row!      │ WHERE id=1    │         │
│  └───────────────┘                     └───────────────┘         │
│                                                                   │
│          CONFLICT! Which value wins? 'A' or 'B'?                 │
│                                                                   │
├──────────────────────────────────────────────────────────────────┤
│                                                                   │
│  CHALLENGE 2: Stale Read                                          │
│  ┌───────────────┐                     ┌───────────────┐         │
│  │     DC1       │                     │     DC2       │         │
│  │   WRITE       │ ─── Sync delay ───► │    READ       │         │
│  │  name = 'A'   │      (not yet)      │   name = ?    │         │
│  └───────────────┘                     └───────────────┘         │
│        T1                                    T2                   │
│                                                                   │
│       Read at T2 might return OLD value (before sync)            │
│                                                                   │
└──────────────────────────────────────────────────────────────────┘
```

### Conflict Resolution Strategies

| Strategy | How It Works |
|----------|--------------|
| **Last Write Wins (LWW)** | Timestamp-based; latest update kept |
| **First Write Wins** | Original value preserved |
| **Version Vectors** | Track causality across nodes |
| **Custom Business Logic** | Application decides resolution |
| **Merge** | Combine both changes intelligently |

### Advantages & Disadvantages

| Advantages | Disadvantages |
|------------|---------------|
| ✅ Full resource utilization | ❌ Complex sync mechanism |
| ✅ No cross-region latency | ❌ Conflict resolution required |
| ✅ Handle more traffic | ❌ Requires Multi-Master DB |
| ✅ Near-instant failover | ❌ Eventual consistency issues |
| ✅ Better for write-heavy apps | ❌ More complex to implement |
| ✅ Both DCs fully active | ❌ Data integrity challenges |

## Architecture Comparison

### Visual Comparison

```
ACTIVE-PASSIVE:
┌─────────────────┐      One-way       ┌─────────────────┐
│    PRIMARY      │ ──────────────────►│    REPLICA      │
│   Read/Write    │       Sync         │   Read-Only     │
│     (Live)      │                    │      (DR)       │
└─────────────────┘                    └─────────────────┘


ACTIVE-ACTIVE:
┌─────────────────┐      Two-way       ┌─────────────────┐
│   PRIMARY 1     │ ◄─────────────────►│   PRIMARY 2     │
│   Read/Write    │       Sync         │   Read/Write    │
│     (Live)      │                    │     (Live)      │
└─────────────────┘                    └─────────────────┘
```

### Detailed Comparison

| Feature | Active-Passive | Active-Active |
|---------|----------------|---------------|
| **Write Nodes** | 1 (Primary only) | Multiple (all) |
| **Read Nodes** | Multiple | Multiple |
| **Sync Direction** | One-way | Bi-directional |
| **DB Support** | All DBs | Multi-Master only |
| **Complexity** | Lower | Higher |
| **Resource Usage** | Partial | Full |
| **Write Latency** | Higher for DR | Low (local) |
| **Failover Time** | Minutes | Near-instant |
| **Conflict Handling** | Not needed | Required |
| **Data Consistency** | Strong | Eventual |

### When to Use Which?

| Scenario | Recommended Architecture |
|----------|--------------------------|
| Using Oracle/MySQL/PostgreSQL | Active-Passive |
| Using Cassandra/NoSQL | Active-Active |
| Read-heavy application | Active-Passive |
| Write-heavy application | Active-Active |
| Need simple architecture | Active-Passive |
| Need maximum performance | Active-Active |
| Global distributed users | Active-Active |
| Strong consistency required | Active-Passive |
| Cost-sensitive | Active-Passive |

## Summary: How HA Goals Are Achieved

| Goal | Active-Passive | Active-Active |
|------|----------------|---------------|
| **99.999% Availability** | ✅ Via failover | ✅ Via redundancy |
| **No SPOF** | ✅ DR takes over | ✅ Any node serves |
| **Data Resilience** | ✅ Replica has data | ✅ All nodes have data |
| **Fast Recovery** | ⚠️ Minutes | ✅ Near-instant |

## Key Interview Points

### Must Remember

1. **Single Node = Single Point of Failure**
2. **Active-Passive**: One Primary + Read-Only Replicas
3. **Active-Active**: Multiple Primaries + Bi-directional Sync
4. **SQL DBs** (Oracle, MySQL, PostgreSQL) → NOT Multi-Master
5. **NoSQL DBs** (Cassandra, etc.) → Support Multi-Master
6. **Failover Gap**: Active-Passive has 10-15 min gap during failover
7. **Conflict Resolution**: Critical challenge in Active-Active

### Common Follow-up Questions

| Question | Key Points |
|----------|------------|
| How does failover work? | Health checks, promotion, traffic rerouting |
| How to handle conflicts? | LWW, version vectors, custom logic |
| What about consistency? | Active-Passive: Strong; Active-Active: Eventual |
| RTO and RPO? | Active-Passive: Higher; Active-Active: Lower |
