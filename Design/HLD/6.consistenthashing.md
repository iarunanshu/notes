# Consistent Hashing - Complete Notes

## Overview
Consistent Hashing is a technique used to solve the rebalancing problem in distributed systems where nodes (servers) are dynamic. It's essential for load balancing and horizontal sharding scenarios.

---

## 1. What is Hashing?

### Basic Concept
```
Key (e.g., "Shreyansh") → Hash Function → Fixed Length Value (e.g., 20)
```

**Hash Function Properties:**
- Takes **arbitrary length** input
- Produces **fixed length** output

### Mod Hashing Technique
Used to map hash values to a fixed-size hash table.

```
Index = Hash(Key) % Size of Hash Table
```

**Example:**
- Hash table size: 6 (indices 0-5)
- Hash("Shreyansh") = 20
- Index = 20 % 6 = 2
- Data stored at index 2

---

## 2. Problem with Traditional Hashing

### When Hashing Works Well
✅ When the **size is fixed**

### When Hashing Fails
❌ When the **size is dynamic** (nodes can be added/removed)

### Use Cases Where Size is Dynamic

| Use Case | Description |
|----------|-------------|
| **Load Balancing** | Application servers can increase/decrease |
| **Horizontal Sharding** | Database servers can be added/removed |

---

## 3. The Rebalancing Problem

### Scenario: Load Balancing Example

**Initial Setup (3 servers):**
```
Load Balancer uses: Hash(request) % 3

Request for "Shreyansh" → Hash % 3 = Node 1
Request for "Rahul" → Hash % 3 = Node 2  
Request for "Shivam" → Hash % 3 = Node 3
```

**After Adding a New Server (4 servers):**
```
Load Balancer now uses: Hash(request) % 4

Request for "Shreyansh" → Hash % 4 = Node 2 (was Node 1!)
Request for "Rahul" → Hash % 4 = Node 3 (was Node 2!)
```

### The Problem
| Issue | Impact |
|-------|--------|
| Modulus changes | All hash calculations change |
| Data location changes | Requests go to wrong nodes |
| **Rebalancing required** | Millions of entries need to be moved |

### What is Rebalancing?
Moving data from old node locations to new node locations when the modulus changes.

**Example:**
- "Shreyansh" data was in Node 1
- After adding server, calculation points to Node 2
- Must move "Shreyansh" data from Node 1 → Node 2
- **Scale:** Could be millions of DB entries!

---

## 4. What is Consistent Hashing?

### Purpose
**Minimize rebalancing** when nodes are added or removed.

### Goal
When a node is added or removed:
```
Rebalancing = (1/n) × 100% of total keys
```
Where **n** = number of active nodes

### Comparison

| Scenario | Traditional Hashing | Consistent Hashing |
|----------|--------------------|--------------------|
| Node added/removed | Rebalance ALL keys | Rebalance only ~1/n% of keys |
| 1000 keys, 4 nodes | ~1000 keys moved | ~250 keys moved |

---

## 5. How Consistent Hashing Works

### Step 1: Create a Virtual Ring

```
        0
    11     1
  10         2
  9           3
    8       4
      7   5
        6
```

- Ring represents positions 0 to (size-1)
- After the last position, it wraps back to 0
- Acts like a **circular queue**

### Step 2: Place Servers on the Ring

Apply hash function to server identifiers:

```
Hash(Server1) % 12 = Position on ring
Hash(Server2) % 12 = Position on ring
Hash(Server3) % 12 = Position on ring
```

**Example Placement:**
```
           0
       11     1
     10         2 ← S1
     9           3 ← S3
       8       4
         7   5
           6 ← S2
```

### Step 3: Place Keys on the Ring

Apply same hash function to keys:

```
Hash(Key1) % 12 = Position on ring
Hash(Key2) % 12 = Position on ring
... and so on
```

### Step 4: Determine Key-to-Server Mapping

**Rule: Move CLOCKWISE from key position until you hit a server**

```
           0
       11  K1 1
     10         2 ← S1 (handles K1, K2)
     9    K7     3 ← S3 (handles K3, K4, K6)
       8  K5   4 K2
     K6  7   5 K3
           6 ← S2 (handles K5, K7)
              K4
```

**Mapping Result:**
| Key | Clockwise → | Server |
|-----|-------------|--------|
| K1 | → | S1 |
| K2 | → | S1 |
| K3 | → | S3 |
| K4 | → | S3 |
| K5 | → | S2 |
| K6 | → | S3 |
| K7 | → | S2 |

---

## 6. Adding a Server (Minimal Rebalancing)

### Before: 3 Servers
```
S1 handles: K1, K2
S2 handles: K5, K7
S3 handles: K3, K4, K6
```

### After: Adding S4 at position 8
```
           0
       11  K1 1
     10         2 ← S1
     9    K7     3 ← S3
    S4→ 8  K5   4 K2
     K6  7   5 K3
           6 ← S2
              K4
```

### New Mapping
| Key | Before | After | Changed? |
|-----|--------|-------|----------|
| K1 | S1 | S1 | ❌ No |
| K2 | S1 | S1 | ❌ No |
| K3 | S3 | S3 | ❌ No |
| K4 | S3 | S3 | ❌ No |
| K5 | S2 | S2 | ❌ No |
| K6 | S3 | S3 | ❌ No |
| K7 | S2 | **S4** | ✅ Yes |

**Result:** Only K7 needs rebalancing! (1 out of 7 keys)

---

## 7. Removing a Server (Minimal Rebalancing)

### Scenario: Remove S2 (at position 6)

### Before Removal
```
S2 handles: K5, K7
```

### After Removal
- K5 and K7 move clockwise to next server
- Next server is S3

### New Mapping
| Key | Before | After | Changed? |
|-----|--------|-------|----------|
| K5 | S2 | **S3** | ✅ Yes |
| K7 | S2 | **S3** | ✅ Yes |
| Others | - | - | ❌ No |

**Result:** Only keys that were on S2 need rebalancing!

---

## 8. Problem: Uneven Distribution

### The Issue
Hash function might place servers close together:

```
           0
       11     1
     10         2 ← S1
     9           3 ← S2
       8       4 ← S3
         7   5
           6
```

### Result of Poor Distribution
```
Keys: K1, K2, K3, K4, K5, K6 all positioned between 5-11

Moving clockwise from each key → ALL hit S1 first!
```

| Server | Keys Handled |
|--------|--------------|
| S1 | K1, K2, K3, K4, K5, K6 (ALL!) |
| S2 | None |
| S3 | None |

**Problem:** Load is NOT equally distributed!

---

## 9. Solution: Virtual Nodes (Replicas)

### Concept
**Replicate each server at multiple positions on the ring**

### How It Works

**Original positions (from hash function):**
- S1 → Position 2
- S2 → Position 3
- S3 → Position 4

**Add virtual replicas:**
```
Server 1: Positions 2, 8, 11
Server 2: Positions 3, 6, 10
Server 3: Positions 4, 1, 9
```

### Visualization with Virtual Nodes
```
           0
    S3→ 11     1 ← S3
  S2→ 10         2 ← S1
    S3→ 9         3 ← S2
     S1→ 8       4 ← S3
         7   5
           6 ← S2
```

### Result: Better Distribution

Now keys get distributed more evenly:

| Key Position | Clockwise → | Server |
|--------------|-------------|--------|
| K1 (at 0) | → | S3 (at 1) |
| K2 (at 5) | → | S2 (at 6) |
| K3 (at 7) | → | S1 (at 8) |
| K4 (at 10) | → | S3 (at 11) |

**More replicas = Better distribution**

### How Many Replicas?

Create enough replicas to achieve:
```
Rebalancing ≈ (1/n) × 100% of total keys
```

When this percentage is achieved, distribution is balanced.

---

## 10. Summary: Consistent Hashing Algorithm

### Setup Phase
```
1. Create virtual ring (0 to max_size-1)
2. For each server:
   - Calculate hash position
   - Add virtual replicas at random positions
3. For each key:
   - Calculate hash position
   - Move clockwise to find assigned server
```

### On Server Addition
```
1. Calculate new server's position(s)
2. Only keys between new server and its counter-clockwise neighbor need rebalancing
3. Move those keys to new server
```

### On Server Removal
```
1. Identify keys on removed server
2. Reassign those keys to next clockwise server
3. Only those keys need rebalancing
```

---

## 11. When to Use Consistent Hashing

### Use Consistent Hashing When:
| Condition | Example |
|-----------|---------|
| Nodes are **dynamic** | Servers can be added/removed |
| Need **equal load distribution** | Traffic/data spread evenly |
| Want **minimal rebalancing** | Avoid moving all data |

### Examples:
- ✅ Load Balancing (app servers scale up/down)
- ✅ Horizontal Sharding (DB nodes change)
- ✅ Distributed Caching (cache nodes vary)
- ✅ CDN Edge Servers

### Don't Need Consistent Hashing When:
- ❌ Number of nodes is **fixed** and **never changes**
- ❌ Simple static hash table scenarios

## 12. Quick Reference

### Traditional Hashing vs Consistent Hashing

| Aspect | Traditional | Consistent |
|--------|-------------|------------|
| Node change impact | All keys affected | ~1/n keys affected |
| Rebalancing cost | High (O(K)) | Low (O(K/n)) |
| Dynamic nodes | ❌ Poor | ✅ Excellent |
| Implementation | Simple | More complex |
| Use case | Static systems | Distributed systems |

### Key Formulas

**Rebalancing on node change:**
```
Keys to rebalance = Total Keys / Number of Nodes
Percentage = (1/n) × 100%
```

### Key Terms

| Term | Definition |
|------|------------|
| **Virtual Ring** | Circular structure representing hash space |
| **Clockwise Rule** | Keys assigned to first server found moving clockwise |
| **Virtual Nodes** | Replicas of servers at multiple ring positions |
| **Rebalancing** | Moving data when node topology changes |

## 13. Interview Quick Points

1. **What problem does consistent hashing solve?**
    - Minimizes rebalancing when nodes are added/removed in distributed systems

2. **How does it work?**
    - Uses virtual ring, places servers and keys using hash function
    - Keys assigned to first server found moving clockwise

3. **What's the rebalancing improvement?**
    - Traditional: 100% keys may need rebalancing
    - Consistent: Only ~1/n% keys need rebalancing

4. **What's the virtual nodes solution for?**
    - Solves uneven distribution problem
    - Ensures load is spread evenly across all servers

5. **Where is it used?**
    - Load balancers, horizontal sharding, distributed caches, CDNs
