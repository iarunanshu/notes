# Design Rate Limiter - Complete Notes

## Overview
Rate limiter is crucial for protecting APIs from abuse and ensuring fair resource usage. It's a very common interview question for backend engineers.

---

## Why Rate Limiter?

### The DDoS Attack Problem

```
                    Attacker
                       │
           ┌──────────┬┴┬──────────┐
           │          │ │          │
           ▼          ▼ ▼          ▼
    ┌──────────────────────────────────┐
    │            SERVER                 │
    │  ┌─────────┐  ┌─────────┐        │
    │  │  RAM    │  │  Disk   │        │
    │  │(Limited)│  │(Limited)│        │
    │  └─────────┘  └─────────┘        │
    └──────────────────────────────────┘
                    │
            Resources Exhausted
                    │
                    ▼
              SERVER DOWN
                    │
                    ▼
    ┌───────────────────────────────┐
    │  Genuine User Request DENIED  │
    └───────────────────────────────┘
```

**DDoS Attack:** Attacker sends thousands/lakhs of requests per second, exhausting server resources (RAM, disk, CPU), causing the server to crash and deny genuine users.

**Solution:** Rate Limiter - Controls the number of requests a client can make in a given time period.

---

## Five Rate Limiting Algorithms

| # | Algorithm | Data Structure | Key Feature |
|---|-----------|----------------|-------------|
| 1 | Token Bucket | Counter | Flexible burst handling |
| 2 | Leaking Bucket | Queue | Constant processing rate |
| 3 | Fixed Window Counter | Counter + Time Window | Simple implementation |
| 4 | Sliding Window Log | Timestamp Log | Accurate but memory heavy |
| 5 | Sliding Window Counter | Counter + Sliding Window | Best of both worlds |

## 1. Token Bucket Algorithm

### Concept

```
                    Refiller
                    (Adds tokens periodically)
                        │
                        ▼
         ┌──────────────────────────┐
         │     TOKEN BUCKET         │
         │  ┌───┐ ┌───┐ ┌───┐ ┌───┐│
         │  │ T │ │ T │ │ T │ │ T ││ ← Capacity: 4 tokens
         │  └───┘ └───┘ └───┘ └───┘│
         └──────────────────────────┘
                        │
              Request consumes 1 token
                        │
                        ▼
                   [Process Request]
```

### Key Properties

| Property | Description | Example |
|----------|-------------|---------|
| **Capacity** | Max tokens bucket can hold | 4 tokens |
| **Refill Rate** | Tokens added per time unit | 2 tokens/minute |
| **Overflow** | Extra tokens discarded | If capacity=4, adding to full bucket = overflow |

### Flow Diagram

```
        Request Arrives
              │
              ▼
     ┌────────────────┐
     │  Token > 0 ?   │
     └───────┬────────┘
             │
      ┌──────┴──────┐
      │             │
     YES            NO
      │             │
      ▼             ▼
┌──────────┐   ┌──────────┐
│ Consume  │   │  DENY    │
│ 1 Token  │   │ Request  │
│ Process  │   │ HTTP 429 │
└──────────┘   └──────────┘
```

### Implementation Example

**Rule:** Each user can make 3 requests per minute for tweet/post API

```
User: user_1
┌─────────────────────────────────────────────────┐
│ Time      │ Action      │ Counter │ Result      │
├───────────┼─────────────┼─────────┼─────────────┤
│ 10:00:01  │ POST tweet  │ 3 → 2   │ ✅ Allowed  │
│ 10:00:25  │ POST tweet  │ 2 → 1   │ ✅ Allowed  │
│ 10:00:35  │ POST tweet  │ 1 → 0   │ ✅ Allowed  │
│ 10:00:45  │ POST tweet  │ 0       │ ❌ Denied   │
│ 10:01:00  │ Refill      │ 0 → 2   │ +2 tokens   │
│ 10:01:10  │ POST tweet  │ 2 → 1   │ ✅ Allowed  │
│ 10:02:00  │ Refill      │ 1 → 3   │ +2 (cap=3)  │
└─────────────────────────────────────────────────┘
```

### Pros & Cons

| Pros | Cons |
|------|------|
| Simple to implement | Memory needed per user/API |
| Allows burst traffic | Need to tune capacity & refill rate |
| Flexible configuration | |

## 2. Leaking Bucket Algorithm

### Concept

```
        Incoming Requests
        (Variable Rate)
              │
              ▼
    ┌───────────────────┐
    │   QUEUE (FIFO)    │
    │ ┌───┬───┬───┬───┐ │
    │ │R1 │R2 │R3 │R4 │ │ ← Fixed Capacity
    │ └───┴───┴───┴───┘ │
    └─────────┬─────────┘
              │
              ▼
         Leak (Process)
         @ Constant Rate
              │
              ▼
         [Server]
```

### Flow Diagram

```
        Request Arrives
              │
              ▼
     ┌────────────────┐
     │  Queue Full ?  │
     └───────┬────────┘
             │
      ┌──────┴──────┐
      │             │
      NO           YES
      │             │
      ▼             ▼
┌──────────┐   ┌──────────┐
│  Add to  │   │  DENY    │
│  Queue   │   │ Request  │
│  (FIFO)  │   │ HTTP 429 │
└──────────┘   └──────────┘
      │
      ▼
┌──────────────────┐
│ Process at       │
│ Constant Rate    │
└──────────────────┘
```

### Pros & Cons

| Pros | Cons |
|------|------|
| Smooth output rate | Not suitable for burst traffic |
| Simple using Queue | Old requests may starve new ones |
| Memory efficient | Fixed processing rate |

### When NOT to Use

**Scenario:** Amazon Prime Video
- Day time: Low traffic
- Evening/Night: High traffic (burst)

Leaking bucket processes at constant rate → During peak, new requests wait while old requests in queue are processed → Poor user experience

**Ask interviewer:** Does the application need to handle traffic bursts?

## 3. Fixed Window Counter Algorithm

### Concept

```
Time: ──────────────────────────────────────────────────►
      │    Window 1   │    Window 2   │    Window 3   │
      │   (5 min)     │    (5 min)    │    (5 min)    │
      │ Counter: 3    │  Counter: 3   │  Counter: 3   │
      │               │               │               │
    08:00          08:05           08:10           08:15
```

### How It Works

```
Rule: 3 requests per 5-minute window

Window: 08:00 - 08:05
┌─────────────────────────────────────────────────┐
│ Time      │ Action   │ Counter │ Result         │
├───────────┼──────────┼─────────┼────────────────┤
│ 08:01:00  │ Request  │ 3 → 2   │ ✅ Allowed     │
│ 08:02:00  │ Request  │ 2 → 1   │ ✅ Allowed     │
│ 08:03:00  │ Request  │ 1 → 0   │ ✅ Allowed     │
│ 08:04:00  │ Request  │ 0       │ ❌ Denied      │
│ 08:05:00  │ New Win  │ Reset→3 │ New window     │
│ 08:05:30  │ Request  │ 3 → 2   │ ✅ Allowed     │
└─────────────────────────────────────────────────┘
```

### ⚠️ Major Disadvantage: Edge Case Problem

```
Rule: 3 requests per 5-minute window

Window 1: 08:05 - 08:10     Window 2: 08:10 - 08:15
        │                            │
        │    ●●●                ●●●  │
        │    └─┬─┘              └─┬─┘│
        │  3 requests        3 requests
        │  at 08:09:58       at 08:10:01
        │                            │
        └────────────────────────────┘
                     │
                     ▼
        In ~3 seconds: 6 REQUESTS!
        (Should be max 3 per 5 min)
```

**Problem:** At window boundaries, double the allowed requests can pass through!

## 4. Sliding Window Log Algorithm

### Concept

Instead of fixed windows, maintain a **log of timestamps** that slides with time.

```
Rule: 3 requests per minute

Sorted Timestamp Log:
┌────────────────────────────────┐
│ 10:00:10 │ 10:00:20 │ 10:00:40 │
└────────────────────────────────┘
     │           │           │
   Request 1  Request 2  Request 3
```

### How It Works

```
Rule: 3 requests per minute

┌──────────────────────────────────────────────────────────┐
│ Time      │ Action          │ Log                │ Result│
├───────────┼─────────────────┼────────────────────┼───────┤
│ 10:00:10  │ Request         │ [10:00:10]         │ ✅    │
│           │                 │ Size: 1            │       │
├───────────┼─────────────────┼────────────────────┼───────┤
│ 10:00:20  │ Request         │ [10:00:10,         │ ✅    │
│           │                 │  10:00:20]         │       │
│           │                 │ Size: 2            │       │
├───────────┼─────────────────┼────────────────────┼───────┤
│ 10:00:40  │ Request         │ [10:00:10,         │ ✅    │
│           │                 │  10:00:20,         │       │
│           │                 │  10:00:40]         │       │
│           │                 │ Size: 3            │       │
├───────────┼─────────────────┼────────────────────┼───────┤
│ 10:00:45  │ Request         │ Log still in       │ ❌    │
│           │ (within 1 min)  │ 1-min window       │       │
│           │                 │ Size: 3 (full)     │       │
├───────────┼─────────────────┼────────────────────┼───────┤
│ 10:01:20  │ Request         │ Remove 10:00:10    │ ✅    │
│           │ (window slides) │ [10:00:20,         │       │
│           │                 │  10:00:40,         │       │
│           │                 │  10:01:20]         │       │
└──────────────────────────────────────────────────────────┘
```

### Window Sliding Logic

```
New Request at time T:
1. Remove all timestamps < (T - window_size)
2. Check if log size < limit
3. If yes: Add timestamp, allow request
4. If no: Deny request (but still log for tracking)
```

### Pros & Cons

| Pros | Cons |
|------|------|
| Accurate rate limiting | High memory usage |
| Solves edge case problem | Logs even denied requests |
| No burst at boundaries | Storage grows with traffic |

## 5. Sliding Window Counter Algorithm

### Concept

Combines **Fixed Window Counter** + **Sliding Window Log**

- Uses counters (memory efficient like Fixed Window)
- Uses sliding concept (accurate like Sliding Window Log)

### How It Works

```
Rule: 5 requests per minute

      Previous Window          Current Window
      (08:01 - 08:02)         (08:02 - 08:03)
    ┌─────────────────┐     ┌─────────────────┐
    │ Counter: 6      │     │ Counter: 2      │
    │ (6 requests)    │     │ (2 requests)    │
    └────────┬────────┘     └────────┬────────┘
             │                       │
             └───────────┬───────────┘
                         │
              Current Sliding Window
              (spanning both windows)
```

### Formula

```
Requests in Sliding Window = 
    Requests in Current Window + 
    (Requests in Previous Window × Overlap Percentage)

Overlap Percentage = Time spent in previous window / Window size
```

### Example Calculation

```
Window Size: 60 seconds
Previous Window: 6 requests
Current Window: 2 requests
Current position: 10 seconds into current window

Overlap with previous window = (60 - 10) / 60 = 50/60 = 0.83

Sliding Window Count = 2 + (6 × 0.83)
                     = 2 + 5
                     = 7 requests

If limit is 5: Request DENIED
```

### Visual Example

```
Time: ─────────────────────────────────────────────────►
      │         Previous           │        Current         │
      │         Window             │        Window          │
      │  ●  ●  ●  ●  ●  ●         │    ●  ●                │
      │  (6 requests)              │   (2 requests)         │
      │                            │                        │
    08:01                       08:02                    08:03
                                   │
                               ◄───┼───────────────────►
                                   │  Sliding Window
                                   │  (1 minute from now)
                                   │
                              Calculates: How many requests
                              are in this sliding window?
```

### Pros & Cons

| Pros | Cons |
|------|------|
| Memory efficient | Not 100% accurate (approximation) |
| Solves boundary issue | Assumes even distribution |
| Smooth rate limiting | |

## Algorithm Comparison

| Algorithm | Memory | Accuracy | Burst Handling | Use Case |
|-----------|--------|----------|----------------|----------|
| Token Bucket | Medium | Good | ✅ Excellent | APIs needing burst |
| Leaking Bucket | Low | Good | ❌ Poor | Constant rate needed |
| Fixed Window | Low | ⚠️ Edge issues | Medium | Simple use cases |
| Sliding Window Log | High | ✅ Excellent | Good | Strict rate limiting |
| Sliding Window Counter | Medium | Good | Good | Balanced approach |

## System Design

### Where to Place Rate Limiter?

```
Option 1: Client-Side              Option 2: Server-Side (API Gateway)

┌────────┐    ┌──────────────┐     ┌────────┐    ┌─────────────┐    ┌────────┐
│ Client │───►│Rate Limiter  │     │ Client │───►│API Gateway  │───►│ Server │
│        │    │(in client)   │     │        │    │+Rate Limiter│    │        │
└────────┘    └──────────────┘     └────────┘    └─────────────┘    └────────┘
                    │
                    ▼
              ┌──────────┐
              │  Server  │
              └──────────┘

❌ Not recommended              ✅ Recommended
(Client can bypass)            (Centralized control)
```

### High-Level Architecture

```
                           ┌─────────────────────────────────────┐
                           │         CONFIG FILES                │
                           │  • Rate limits per API              │
                           │  • Window sizes                     │
                           │  • Algorithm selection              │
                           └───────────────┬─────────────────────┘
                                           │
                                           ▼
                                    ┌──────────────┐
                                    │    CACHE     │
                                    │(Config Cache)│
                                    └──────┬───────┘
                                           │
┌────────┐         ┌─────────────────────────────────────┐         ┌────────┐
│        │         │           RATE LIMITER              │         │        │
│ Client │────────►│  ┌─────────────────────────────┐   │────────►│ Server │
│        │         │  │    Rate Limiting Logic       │   │         │        │
│        │◄────────│  │    (Token/Leaking/etc.)      │   │◄────────│        │
└────────┘         │  └─────────────────────────────┘   │         └────────┘
    │              └───────────────┬─────────────────────┘
    │                              │
    │                              ▼
    │                       ┌──────────────┐
    │                       │    REDIS     │
    │                       │  • Counters  │
    │                       │  • Queues    │
    │                       │  • Logs      │
    │                       └──────────────┘
    │
    │ HTTP 429 (Too Many Requests)
    │ when rate limit exceeded
```

### Components

| Component | Purpose |
|-----------|---------|
| **Rate Limiter** | Hosts the rate limiting algorithm |
| **Redis** | In-memory store for counters/queues (fast access) |
| **Config Files** | Store rules (limits, windows, per API/user) |
| **Cache** | Cache config files (read on startup) |

### Why Redis?

| Requirement | Why Redis? |
|-------------|------------|
| Fast read/write | In-memory storage |
| Counter operations | Atomic increment/decrement |
| Queue support | Built-in list data structure |
| Centralized | Single source of truth |

## Distributed Rate Limiter

### Challenge: Multiple Rate Limiter Instances

```
                    ┌─────────────────┐
                    │     REDIS       │
                    │   (Centralized) │
                    └────────┬────────┘
                             │
        ┌────────────────────┼────────────────────┐
        │                    │                    │
        ▼                    ▼                    ▼
┌───────────────┐   ┌───────────────┐   ┌───────────────┐
│Rate Limiter 1 │   │Rate Limiter 2 │   │Rate Limiter 3 │
└───────────────┘   └───────────────┘   └───────────────┘
        ▲                    ▲                    ▲
        │                    │                    │
        └────────────────────┴────────────────────┘
                             │
                          Client
            (Can hit any rate limiter)
```

### Race Condition Problem

```
Time T1:
┌─────────────────┐     ┌─────────────────┐
│ Rate Limiter 1  │     │ Rate Limiter 2  │
│ Read counter: 2 │     │ Read counter: 2 │
└────────┬────────┘     └────────┬────────┘
         │                       │
         │     Both read 2       │
         │     Both allow        │
         │     Both decrement    │
         │                       │
         ▼                       ▼
    Counter should be 0, but race condition
    may cause inconsistency
```

### Solution: Atomic Operations

```
Options:
1. Redis Lua Scripts (atomic operations)
2. Redis INCR/DECR (atomic)
3. Distributed locks (more latency)

Trade-off: Atomicity vs Latency
```

## HTTP Response Codes

| Code | Meaning | When Used |
|------|---------|-----------|
| **200** | OK | Request allowed and processed |
| **429** | Too Many Requests | Rate limit exceeded |

### Response Headers (Best Practice)

```
HTTP/1.1 429 Too Many Requests
Content-Type: application/json
X-RateLimit-Limit: 100
X-RateLimit-Remaining: 0
X-RateLimit-Reset: 1640000000
Retry-After: 60

{
    "error": "Rate limit exceeded",
    "message": "Too many requests. Please retry after 60 seconds."
}
```

## Interview Tips

### Questions to Ask Interviewer

1. **Scope:** Client-side or server-side rate limiting?
2. **Scale:** Single server or distributed?
3. **Granularity:** Per user? Per API? Per IP?
4. **Traffic pattern:** Steady or bursty?
5. **Accuracy:** Strict limiting or approximate OK?

### Algorithm Selection Guide

| Scenario | Recommended Algorithm|
|----------|----------------------|
| Need to allow bursts | Token Bucket |
| Need constant processing rate | Leaking Bucket |
| Simple implementation, OK with edge issues | Fixed Window Counter |
| Strict accuracy needed | Sliding Window Log |
| Balanced approach | Sliding Window Counter |

### Key Points to Mention

1. **HTTP 429** - Standard response for rate limiting
2. **Redis** - For centralized counter storage
3. **Atomicity** - Critical for distributed systems
4. **Config-driven** - Rules should be configurable
5. **Headers** - Return rate limit info in response headers
