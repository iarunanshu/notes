# Back-of-the-Envelope Estimation - Complete Notes

## Overview
Back-of-the-envelope estimation is a critical first step in system design interviews. It helps justify design decisions with approximate calculations before diving into the actual architecture.

---

## 1. What is Back-of-the-Envelope Estimation?

### Definition
A rough, high-level calculation to estimate system requirements (servers, storage, bandwidth) that **drives design decisions**.

### Why It's Needed

Without estimation, interviewers may ask:
- Do you really need a load balancer?
- Do you really need CDN?
- How many servers will you need?
- What's the storage capacity required?
- Do you need caching? What size?

### Purpose
| Goal | Description |
|------|-------------|
| Justify components | Prove why load balancer, cache, etc. are needed |
| Right-size resources | Avoid over/under utilization |
| Show system thinking | Demonstrate you consider constraints |
| Drive architecture | Numbers inform design choices |

---

## 2. Key Considerations

### ⚠️ Important Rules

| Rule | Explanation |
|------|-------------|
| **Rough/T-shirt sizing** | Not exact numbers, just estimates |
| **Don't spend too much time** | < 10 minutes maximum |
| **Keep assumptions simple** | Use round numbers (10M, 100M, not 273M) |
| **Won't change design much** | 95% of time, you'll need scalable design anyway |

### Pro Tip
> Always ask the interviewer: "Do you want me to do back-of-the-envelope estimation?"
> If yes, keep it under 10 minutes.

## 4. Quick Calculation Formula ⭐

### The Magic Formula
```
X million users × Y MB per user = X×Y TB
```

### How It Works
```
Million = 10^6 (6 zeros)
MB = 10^6 bytes (6 zeros)
Combined = 10^12 = TB (12 zeros)
```

### Examples

**Example 1:**
```
5 million users × 2 KB each = ?

Million = 6 zeros
KB = 3 zeros
Total = 9 zeros = GB

Answer: 5 × 2 = 10 GB
```

**Example 2:**
```
100 million users × 5 MB each = ?

Million = 6 zeros
MB = 6 zeros  
Total = 12 zeros = TB

Answer: 100 × 5 = 500 TB
```

## 6. Complete Example: Facebook Estimation

### Step 1: Traffic Estimation

#### Basic Assumptions
| Metric | Value |
|--------|-------|
| Total Users | 1 billion |
| Daily Active Users (DAU) | 25% = **250 million** |
| Read operations/user/day | 5 |
| Write operations/user/day | 2 |
| Total queries/user/day | 7 |

#### Calculate Queries Per Second (QPS)
```
Daily queries = 250 million users × 7 queries
             = 1,750 million queries/day

Seconds per day ≈ 100,000 (rounded from 86,400)

QPS = 1,750 million ÷ 100,000
    = 17,500 ≈ 18K queries/second
```

**Result: ~18,000 QPS**

---

### Step 2: Storage Estimation

#### Assumptions
| Item | Value |
|------|-------|
| Posts per user per day | 2 |
| Characters per post | 250 |
| Bytes per character | 2 |
| Users uploading images | 10% of DAU |
| Image size | 300 KB |

#### Calculate Post Storage

```
One post = 250 characters × 2 bytes = 500 bytes
Two posts = 1,000 bytes = 1 KB

Daily post storage = 250 million users × 1 KB
                   = 250 million KB

Million (6) + KB (3) = 9 zeros = GB

Result: 250 GB per day for posts
```

#### Calculate Image Storage

```
Users uploading = 10% of 250 million = 25 million
Image size = 300 KB

Daily image storage = 25 million × 300 KB

Million (6) + KB (3) = 9 zeros = GB
25 × 300 = 7,500 GB ≈ 7.5 TB

Result: ~8 TB per day for images
```

#### Calculate 5-Year Storage

```
5 years ≈ 2,000 days

Posts: 250 GB × 2,000 = 500,000 GB = 500 TB
Images: 8 TB × 2,000 = 16,000 TB = 16 PB
```

| Data Type | Daily | 5 Years |
|-----------|-------|---------|
| Posts | 250 GB | 500 TB |
| Images | 8 TB | 16 PB |

---

### Step 3: RAM/Cache Estimation

#### Assumptions
| Item | Value |
|------|-------|
| Cached data per user | Last 5 posts |
| Post size | 500 bytes |
| Cache per user | 5 × 500 = 2,500 bytes ≈ 3 KB |

#### Calculate Total Cache Needed

```
Cache needed = 250 million users × 3 KB

Million (6) + KB (3) = 9 zeros = GB
250 × 3 = 750 GB

Result: 750 GB RAM for caching
```

#### Calculate Cache Machines Needed

```
If one machine holds 75 GB RAM:
Machines needed = 750 GB ÷ 75 GB = 10 machines
```

**Result: 10 cache machines**

---

### Step 4: Server Estimation

#### Assumptions
| Item | Value |
|------|-------|
| Latency target | 500ms (95th percentile) |
| Threads per server | 50 |
| Requests per thread per second | 2 (since 500ms each) |

#### Calculate Servers Needed

```
Requests per server per second = 50 threads × 2 = 100

Total QPS = 18,000
Servers needed = 18,000 ÷ 100 = 180 servers
```

**Result: ~180 application servers**

---

### Step 5: CAP Theorem Trade-off

#### CAP Theorem Review
```
       C (Consistency)
      / \
     /   \
    /     \
   A-------P
(Availability) (Partition Tolerance)
```

You can only achieve 2 out of 3 at any time:
- **CA** - Consistency + Availability
- **CP** - Consistency + Partition Tolerance
- **AP** - Availability + Partition Tolerance

#### Facebook's Choice: AP (Availability + Partition Tolerance)

| Property | Needed? | Reason |
|----------|---------|--------|
| **Availability** | ✅ Yes | System must always respond |
| **Partition Tolerance** | ✅ Yes | Must work despite network failures |
| **Consistency** | ❌ Trade-off | Eventual consistency is acceptable |

**Justification:**
- Users expect Facebook to always be available
- Network partitions will happen in distributed systems
- Slight delay in seeing latest posts is acceptable

## 10. Key Interview Tips

1. **Always ask first** - "Should I do back-of-the-envelope estimation?"

2. **Use round numbers** - 250 million, not 247 million

3. **Show your work** - Talk through calculations

4. **Don't obsess over accuracy** - T-shirt sizing is fine

5. **Connect to design** - "Based on 18K QPS, we need load balancing"

6. **Know your zeros** - Memorize the conversion table

7. **State assumptions clearly** - "Assuming 2 bytes per character..."

8. **Include CAP trade-off** - Shows architectural thinking
