# Scaling from Zero to 1 Million Users - Complete Notes

## Overview
This guide covers 9 steps to scale a system from zero to millions of users, covering key high-level design concepts including CDN, horizontal/vertical scaling, caching, load balancers, and more.

---

## Step 1: Single Server Setup

**What it is:**
- Basic setup typically used in college projects or initial launches
- Everything runs on one server

**Architecture:**
```
Client (Web/Mobile) → Server (Application + Database)
```

**Characteristics:**
- Client directly connects to server
- Application server and database hosted on the same machine
- Suitable for zero or very few users
- No separation of concerns

---

## Step 2: Application and DB Server Separation

**Why needed:**
- To grow application and database independently
- Remove dependency between application and database
- Allow independent scaling

**Architecture:**
```
Client → Mid-Tier Layer (Application Server) → DB Tier (Database Server)
```

**Key Points:**
- **Mid-tier layer**: Hosts application server, runs only business logic
- **DB tier**: Separate server for database
- Both can now scale independently

---

## Step 3: Load Balancer + Multiple App Servers

**Why needed:**
- Single app server has limited request handling capacity (e.g., 1000 requests/minute)
- Need to handle more traffic
- Need security layer

**Architecture:**
```
Client (Web/Mobile) → Load Balancer → Multiple App Servers (App Server 1, 2, 3...)
```

**Load Balancer Functions:**
1. **Traffic Distribution**: Equally divides incoming traffic among app servers
2. **Security**: Acts as a barrier between client and servers
3. **Privacy**: Communicates with app servers via **private IPs**
4. **Failure Handling**: Redirects traffic away from failed servers

**Key Benefits:**
- Clients cannot directly access application servers
- Adds a layer of security
- Enables horizontal scaling of application layer

---

## Step 4: Database Replication (Master-Slave)

**Concept:**
- Master-Slave architecture for database reliability and performance

**Architecture:**
```
App Servers → Master DB (Write Operations)
           → Slave DB(s) (Read Operations)
```

**How it works:**
| Operation | Database |
|-----------|----------|
| CREATE | Master DB |
| UPDATE | Master DB |
| DELETE | Master DB |
| READ | Slave DB(s) |

**Key Features:**
- Multiple slave DBs can exist
- Slaves replicate data from master
- **Failover**: If master fails, one slave gets promoted to master
- **High Availability**: If one DB node fails, others continue serving requests

**Benefits:**
- Handles database failure scenarios
- Distributes read load across slaves
- System remains operational even during partial failures

---

## Step 5: Cache Implementation

**Why needed:**
- DB operations are expensive (network calls)
- Improves performance by reducing DB calls

**Architecture:**
```
App Server → Cache → Database
```

**How it works:**
1. Application first checks cache for data
2. **Cache Hit**: Data found in cache → return immediately
3. **Cache Miss**: Data not in cache → fetch from DB → store in cache → return

**TTL (Time To Live):**
- Defines how long data stays in cache (e.g., 24 hours, 48 hours, 7 days)
- After TTL expires, data is purged
- Fresh requests will miss cache and fetch from DB

**Example:** Redis

**Benefits:**
- Reduced DB calls
- Improved response time
- Better performance

## Step 6: CDN (Content Delivery Network)

**Important Distinction:**
> "CDN does caching, but not everyone who does caching is CDN"

**What is CDN:**
- Network of distributed servers for content delivery
- Caches **static data** (HTML, CSS, videos, images)
- Data that changes infrequently

**The Problem CDN Solves:**

Without CDN (Server in India):
| User Location | Latency |
|---------------|---------|
| India | 1 ms |
| Saudi Arabia | 2 ms |
| USA | 3 ms |
| Japan | 4 ms |

**How CDN Works:**
```
User Request → Nearest CDN Node → (if miss) → Neighbor CDN → (if miss) → Original Server
```

**CDN Placement in Architecture:**
```
Client → CDN → Load Balancer → App Servers
```

**Flow:**
1. Client request goes to CDN first (before load balancer)
2. If CDN has data (cache hit) → return to client
3. If CDN doesn't have data → check neighbor CDN
4. If neighbor doesn't have → request goes to original server

**Data Storage:** By URL (and other methods)

**Benefits:**
| Benefit | Description |
|---------|-------------|
| Performance | Reduced latency for global users |
| Security | Protection against DDoS attacks, bot detection |
| Cost Cutting | Fewer DB servers needed |
| Load Reduction | Less load on original server |

## Step 7: Multiple Data Centers

**Why needed:**
- Geographic distribution of users
- Disaster recovery
- Reduced latency for global users

**Architecture:**
```
Client → Load Balancer → Data Center 1 (India) / Data Center 2 (USA)
```

**Each Data Center Contains:**
- Multiple App Servers
- Master-Slave DB setup
- Cache
- CDN nodes

**Load Balancer Role:**
- Routes requests based on **geo-location**
- Sends to nearest data center
- Handles failover (if India DC fails, traffic goes to USA DC)

**DB Replication:**
- Happens between data centers (India ↔ USA)

**Benefits:**
- Reduced load on single data center
- Geographic redundancy
- Disaster recovery capability

## Step 8: Messaging Queue

**Why needed:**
- Brings **asynchronous nature** to the system
- Handles heavy operations without blocking request threads
- Enables retry mechanisms

**Examples:** RabbitMQ, Kafka

**Architecture:**
```
Producer → Messaging Queue (Topics/Queues) → Subscribers
```

**Use Cases:**
- Sending notifications
- Sending emails
- Web notifications
- Any async operation

**RabbitMQ Terminology:**

```
Producer → Exchange → (Binding) → Queue(s) → Subscriber(s)
```

| Component | Description |
|-----------|-------------|
| Exchange | Receives messages from producer |
| Queue | Stores messages |
| Binding | Links exchange to queue |
| Routing Key | Sent by producer with message |
| Binding Key | Defines which queue receives which messages |

**Exchange Types:**

### 1. Direct Exchange
- Routing key **exactly matches** binding key
- Message goes to specific queue
```
Routing Key: "1" → Binding Key: "1" → Queue 1 ✓
Routing Key: "1" → Binding Key: "2" → Queue 2 ✗
```

### 2. Fan-out Exchange
- Sends message to **ALL queues**
- No filtering
- Subscribers decide whether to process or ignore

### 3. Topic Exchange
- **Wildcard comparison** (not exact match)
- Can send to **more than one queue**
- Pattern matching for routing

**Dead Letter Queue (DLQ):**
- Failed messages go to DLQ
- Enables retry mechanism
- Can fix failure and reprocess

**Benefits:**
- Async processing
- Faster request handling
- Retry capability
- Decoupled architecture

## Step 9: Database Scaling

### Two Types:

### A. Vertical Scaling
**What it is:** Increase capacity of existing DB servers

**How:**
- Increase CPU capability
- Increase RAM capability

**Drawbacks:**
- Has physical limits
- Cannot increase indefinitely
- Eventually hits ceiling

### B. Horizontal Scaling
**What it is:** Add more database nodes

**Implementation:** Sharding

---

## Sharding

### 1. Horizontal Sharding (Row-wise)

**Concept:** Divide rows across multiple tables/shards

**Example:**
| Shard | Data |
|-------|------|
| T1 (Shard 1) | Rows 1-500 |
| T2 (Shard 2) | Rows 501-1000 |

**Logic-based Example:**
| Shard | Criteria |
|-------|----------|
| Shard T1 | Names A-P |
| Shard T2 | Names Q-Z |

### 2. Vertical Sharding (Column-wise)

**Concept:** Divide columns across multiple tables

**Example:**
| Shard | Data |
|-------|------|
| Table 1 | Columns C1-C5 (all rows) |
| Table 2 | Columns C6-C10 (all rows) |

**Note:** Each shard contains ALL rows, just different columns

---

## Sharding Drawbacks

### 1. Uneven Distribution (Hotspot Problem)
- Some shards may fill up faster
- Example: Names starting with 'S' might be more common
- **Solution:** Re-sharding (creating sub-shards)
- Can form a tree structure of shards

### 2. Join Difficulties
- Data is distributed across shards
- Joins become complex/impossible
- **Solution:** Denormalization (avoid joins)

### 3. Resharding Complexity
- **Solution:** Consistent Hashing (separate topic)
