# Design Key-Value Database (DynamoDB) - Complete Notes

## Overview
Key-Value databases like Amazon DynamoDB are used for high-scale applications. Amazon uses DynamoDB internally for features like "Add to Cart."

---

## Goals of Key-Value Database

| Goal | Description |
|------|-------------|
| **Scalability** | Handle millions/billions of users distributed across regions |
| **Decentralization** | System remains operational even when nodes fail |
| **Eventual Consistency** | Data becomes consistent over time (not immediately) |

---

## Six Key Concepts

| Step | Concept | Purpose |
|------|---------|---------|
| 1 | Partition | Achieves Scalability |
| 2 | Replication | Achieves Decentralization |
| 3 | Get/Put Operations | Core database operations |
| 4 | Data Versioning | Handles conflicts |
| 5 | Gossip Protocol | Node communication & failure detection |
| 6 | Merkle Tree | Efficient data synchronization |

## 1. Partition (Consistent Hashing)

### Why Partition?
- Millions/billions of users
- Single hash table on one system cannot scale
- Need to distribute data across multiple servers

### How It Works: Consistent Hashing

#### Virtual Ring Structure
```
            0
        ┌───────┐
    S4 /         \ S1
      │           │ (Range: 1-50)
      │           │
    S3 \         / S2
        └───────┘
      (Range: 101-150)
```

#### Range Assignment
| Server | Range |
|--------|-------|
| S1 | 1 - 50 |
| S2 | 51 - 100 |
| S3 | 101 - 150 |
| S4 | 151 - 200 |

#### Example: Storing Data
```
PUT "car" → Generate Key → Key = 45
45 falls in range 1-50 → S1 handles it

PUT "India" → Generate Key → Key = 175
175 falls in range 151-200 → S4 handles it
```

### Virtual Nodes (Solving Hot Key Problem)

**Problem:** Certain key ranges may get more traffic (hot spots)

**Solution:** Place each server at multiple positions on the ring

```
Before Virtual Nodes:
S1 handles: All keys 1-50 (could be overloaded)

After Virtual Nodes:
S1 appears at positions: 2, 45, 120, 180
Traffic distributed across multiple virtual positions
```

**Result:** ✅ Scalability achieved through partitioning

## 2. Replication

### Why Replication?
- If S1 goes down, data stored on S1 becomes unavailable
- Need backup copies for fault tolerance

### Key Concepts

#### Coordinator
The server whose range contains the key
```
Key = 45 → Range 1-50 → S1 is Coordinator
```

#### Replication Factor (N)
- Default: N = 3 (configurable)
- Means: 3 copies of data (1 coordinator + 2 replicas)

#### How Replication Works
```
1. Key arrives at Coordinator (S1)
2. S1 stores data locally
3. S1 replicates to N-1 servers (clockwise direction)
4. S2 and S3 receive copies
```

```
        S1 (Coordinator)
       /  \
      ↓    ↓
     S2    S3
   (Replica) (Replica)
```

### Preference List

**Definition:** For each key range, maintains list of servers holding that data

```
Key Range: 1-50
Preference List:
┌────────────────────┐
│ 1. S1 (Coordinator)│
│ 2. S2 (Replica)    │
│ 3. S3 (Replica)    │
└────────────────────┘
```

### Replica Selection Criteria
- Choose servers in **different data centers**
- Skip **virtual nodes** of same server
- Move **clockwise** but selection may not be sequential

**Why different data centers?**
- If one data center has fire/outage, other replicas survive

**Result:** ✅ Decentralization achieved through replication

## 3. Get and Put Operations

### The R + W > N Formula

| Variable | Meaning |
|----------|---------|
| **N** | Total number of replicas |
| **W** | Write quorum (successful writes needed before returning success) |
| **R** | Read quorum (successful reads needed before returning data) |

**Rule:** R + W > N ensures strong consistency

### PUT Operation Flow

```
Client                Load Balancer           Consistent Hashing Ring
  │                        │                          │
  │ PUT(key1, "car")       │                          │
  │───────────────────────→│                          │
  │                        │    Route to Coordinator  │
  │                        │─────────────────────────→│
  │                        │                     ┌────┴────┐
  │                        │                     │   S1    │
  │                        │                     │(Coord)  │
  │                        │                     └────┬────┘
  │                        │                          │
  │                        │              ┌───────────┼───────────┐
  │                        │              ↓           │           ↓
  │                        │           ┌──────┐      │      ┌──────┐
  │                        │           │  S2  │      │      │  S3  │
  │                        │           │(Copy)│      │      │(Copy)│
  │                        │           └──────┘      │      └──────┘
  │                        │                         │
  │      Success           │←────────────────────────│
  │←───────────────────────│   (After W responses)   │
```

#### PUT Steps:
1. Generate key from data
2. Find coordinator (server whose range contains key)
3. Coordinator stores data locally
4. Coordinator sends async requests to N-1 replicas
5. Wait for **W** successful responses
6. Return success to client

**Example with W=1, N=3:**
- S1 stores data
- S1 sends to S2 and S3 asynchronously
- As soon as 1 replica confirms → Return success

### GET Operation Flow

```
Client                 S1 (Coordinator)              S2, S3 (Replicas)
  │                        │                              │
  │ GET(key1)              │                              │
  │───────────────────────→│                              │
  │                        │   Request data from all     │
  │                        │─────────────────────────────→│
  │                        │                              │
  │                        │←─────────────────────────────│
  │                        │   (Wait for R responses)    │
  │    Return data         │                              │
  │←───────────────────────│                              │
```

#### GET Steps:
1. Request reaches coordinator
2. Coordinator asks ALL replicas for their data
3. Wait for **R** successful responses
4. Reconcile versions (if different)
5. Return data to client

**Why read from multiple replicas?**
- Different replicas may have different versions
- Need to find and return the latest version

### Load Balancer Types

| Type | Behavior | Latency | Complexity |
|------|----------|---------|------------|
| **Generic** | Routes to any node → Node hops to coordinator | High (extra hop) | Simple |
| **Partition-Aware** | Routes directly to coordinator | Low | Complex |

```
Generic Load Balancer:
Client → LB → S4 → (hop) → S1 (coordinator)

Partition-Aware Load Balancer:
Client → LB → S1 (coordinator) [direct]
```

## 4. Data Versioning (Vector Clocks)

### Why Versioning is Needed

**Problem:** Network failures can cause different servers to have different data

#### Failure Scenario Example:
```
Initial State:
S1, S2, S3 all have: key1 = "car"

Step 1: Update to "cart" → S1 is down
        S2 receives update: key1 = "cart"
        
Step 2: Network partition - S2 can't reach S3

Step 3: Another update to "carM" → Goes to S3
        S3 updates: key1 = "carM"

Result:
S1: key1 = "car" (stale)
S2: key1 = "cart"
S3: key1 = "carM"

Three different versions exist!
```

### Vector Clock Solution

**Definition:** List of (Server, Counter) pairs for each key

**Format:** `[S1:counter1, S2:counter2, ...]`

### Vector Clock Example

#### Step 1: Initial PUT
```
PUT key1 = "car" → S1 (coordinator)

S1 stores: key1 = "car" [S1:1]
Replicates to S2, S3:
  S2: key1 = "car" [S1:1]
  S3: key1 = "car" [S1:1]
```

#### Step 2: Update (S1 still up)
```
PUT key1 = "cart" → S1

S1 updates: key1 = "cart" [S1:2]
S1 can reconcile: [S1:1] < [S1:2] → Discard old
Replicates to S2, S3
```

#### Step 3: Update (S1 down)
```
PUT key1 = "carM" → S1 is down → S2 handles

S2 updates: key1 = "carM" [S1:2, S2:1]
S2 tries to replicate but:
  - S1 is down
  - Network partition to S3
```

#### Step 4: Parallel Update
```
PUT key1 = "carR" → S3 handles

S3 updates: key1 = "carR" [S1:2, S3:1]
```

#### Current State:
```
S1: key1 = "cart"  [S1:2]
S2: key1 = "carM"  [S1:2, S2:1]
S3: key1 = "carR"  [S1:2, S3:1]
```

#### Step 5: GET Request (S1 back up)
```
GET key1 → S1 (coordinator)

S1 asks S2 and S3 for their versions:
  S1 has: "cart"  [S1:2]
  S2 has: "carM"  [S1:2, S2:1]
  S3 has: "carR"  [S1:2, S3:1]

S1 reconciliation:
  - All have [S1:2] base
  - S2 and S3 have additional counters
  - Cannot determine which is latest
  - CONFLICT!
```

### Conflict Resolution

```
S1 returns BOTH versions to client:
  Version 1: "carM" [S1:2, S2:1]
  Version 2: "carR" [S1:2, S3:1]

Client (Add-to-Cart Service):
  - Has conflict resolution algorithm
  - Example: "Last Write Wins"
  - Resolves and sends PUT with merged version

New PUT: key1 = "resolved_value" [S1:3, S2:1, S3:1]
  - Merges all version info
  - Replicates to all servers
```

### Conflict Resolution Algorithms
| Algorithm | Description |
|-----------|-------------|
| Last Write Wins | Latest timestamp wins |
| Application Logic | Business rules decide |
| Merge | Combine conflicting values |

**Result:** ✅ Eventual Consistency achieved

## 5. Gossip Protocol

### Purpose
- How do servers know about each other?
- How do servers detect failures?

### How It Works

```
Each server maintains:
┌─────────────────────────────────┐
│ Membership List                 │
├──────────┬──────────┬──────────┤
│ Server   │ Status   │ Last Seen│
├──────────┼──────────┼──────────┤
│ S1       │ UP       │ 12:00 PM │
│ S2       │ UP       │ 11:58 AM │
│ S3       │ UP       │ 11:59 AM │
│ S4       │ UP       │ 12:00 PM │
└──────────┴──────────┴──────────┘
```

### Heartbeat Mechanism

```
Every T seconds:
  S1 → sends heartbeat + metadata → Random nodes (S3, S4)
  
Metadata includes:
  - "I am alive"
  - "I handle range 1-50"
  - "My counter is X"
```

### Failure Detection

```
If server S2 stops sending heartbeats:

S1 notices: S2 last seen 10 minutes ago
S3 notices: S2 last seen 10 minutes ago
S4 notices: S2 last seen 10 minutes ago

When MULTIPLE servers agree S2 is unresponsive:
  → S2 marked as DOWN
  → Removed from active ring
```

**Key Points:**
- Single server suspicion is not enough
- Multiple servers must agree
- Prevents false positives from temporary network issues

## 6. Merkle Tree

### Purpose
Efficiently identify which data is out of sync between coordinator and replicas

### Problem Without Merkle Tree
```
Coordinator S1: Has 1 million keys (range 1-1000000)
Replica S2: Should have same 1 million keys

Naive approach: Compare all 1 million keys
  → Very slow and expensive!
```

### Merkle Tree Solution

```
                    Root Hash: H(AB)
                    /              \
                   /                \
            H(A) = H(1,2)      H(B) = H(3,4)
              /    \              /    \
             /      \            /      \
           Key1    Key2       Key3    Key4
           (v1)    (v2)       (v3)    (v4)
```

### How Comparison Works

```
Step 1: Compare root hashes
        S1 root: 121
        S2 root: 127
        → Different! Need to dig deeper

Step 2: Compare left subtree
        S1 left: 107
        S2 left: 107
        → Same! Keys 1-2 are in sync ✓

Step 3: Compare right subtree
        S1 right: 100
        S2 right: 80
        → Different! Check keys 3-4

Step 4: Compare individual keys
        Find exactly which keys differ
        → Only sync those specific keys
```

### Efficiency Comparison

| Method | Keys to Check | Time Complexity |
|--------|---------------|-----------------|
| Naive | All N keys | O(N) |
| Merkle Tree | Log(N) comparisons | O(log N) |

**Example:**
- 1 million keys
- Naive: Check 1,000,000 keys
- Merkle Tree: ~20 comparisons to find divergent branch

## CAP Theorem Trade-off

DynamoDB chooses: **AP (Availability + Partition Tolerance)**

| Property | DynamoDB Choice |
|----------|-----------------|
| Consistency | ❌ Sacrificed (Eventual) |
| Availability | ✅ Prioritized |
| Partition Tolerance | ✅ Required |

**Result:**
- May return stale data initially
- Eventually all replicas converge
- System remains available during network partitions

## Interview Quick Points

1. **Why consistent hashing?**
    - Enables horizontal scaling
    - Minimizes data movement when nodes change

2. **What is a coordinator?**
    - Server responsible for a key's range
    - First in preference list

3. **What is W in put operation?**
    - Number of successful replica writes before returning success

4. **What is R in get operation?**
    - Number of successful reads before returning data

5. **How are conflicts resolved?**
    - Vector clocks track versions
    - Client resolves when server can't

6. **How are failures detected?**
    - Gossip protocol with heartbeats
    - Multiple servers must agree

7. **How is data sync verified?**
    - Merkle trees for efficient comparison
