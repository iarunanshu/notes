# Design URL Shortener - Complete Notes

## Overview
URL Shortener is a classic system design interview question. Services like TinyURL, Bitly, and LinkedIn's link shortener convert long URLs into short, manageable links.

---

## 1. What is URL Shortener?

### Basic Concept
```
Long URL: https://www.youtube.com/playlist?list=PLA3GkZPtsafZdT...
Short URL: https://lnkd.in/gY9UKQPX
```

Both URLs redirect to the same destination.

### Core APIs

| API | Method | Description |
|-----|--------|-------------|
| Create Short URL | POST | Input: Long URL → Output: Short URL |
| Redirect | GET | Input: Short URL → Redirect to Long URL |

### Database Schema
```
| short_url | long_url |
|-----------|----------|
| gY9UKQPX  | https://youtube.com/playlist?list=... |
```

## 2. Requirement Analysis

### Step 1: Determine URL Length
**Key Question:** How short should the URL be?
**Answer:** As short as possible

### Step 2: Understand Traffic Requirements

**Given:**
- 10 million URLs per day
- Service should last 100 years

**Calculation:**
```
URLs per year = 10 million × 365 = 3,650 million
URLs for 100 years = 3,650 million × 100 = 365 billion URLs
```

### Step 3: Determine Character Set

**Allowed Characters:**
| Type | Characters | Count |
|------|------------|-------|
| Digits | 0-9 | 10 |
| Lowercase | a-z | 26 |
| Uppercase | A-Z | 26 |
| **Total** | | **62** |

### Step 4: Calculate Required Length

**Formula:** Total combinations = 62^n (where n = number of characters)

| Characters | Combinations | Sufficient? |
|------------|--------------|-------------|
| 6 | 62^6 = ~56 billion | ❌ No (need 365 billion) |
| 7 | 62^7 = ~3.5 trillion | ✅ Yes |

**Result: We need 7 characters**

---

## 3. Hash Generation Methods

### Option 1: Hash Functions (MD5 / SHA-1)

#### MD5
- **Size:** 128-bit = 16 bytes
- **Output:** 32 hexadecimal characters
- **Example:** `a3f2b8c9d4e5f6a7b8c9d0e1f2a3b4c5`

#### SHA-1
- **Size:** 160-bit = 20 bytes
- **Output:** 40 hexadecimal characters

#### Problem with Hash Functions

| Issue | Explanation |
|-------|-------------|
| Length too long | MD5 = 32 chars, SHA-1 = 40 chars (need only 7) |
| Must trim | Take first 7 characters only |
| **Collision risk** | Different URLs may have same first 7 chars |

```
URL1 → Hash: "abcdefg12345678..." → Trimmed: "abcdefg"
URL2 → Hash: "abcdefgXYZ98765..." → Trimmed: "abcdefg" ← COLLISION!
```

**Conclusion:** ❌ Hash functions NOT recommended due to collision risk

---

### Option 2: Base62 Encoding ✅ (Recommended)

#### What is Base62?
Converting a decimal number to base 62 using 62 characters (0-9, a-z, A-Z)

#### Base62 Character Mapping
```
0-9   → 0-9   (values 0-9)
a-z   → 10-35 (values 10-35)
A-Z   → 36-61 (values 36-61)
```

#### Conversion Example
```
Decimal 1000 → Base62

1000 ÷ 62 = 16 remainder 8
16 ÷ 62 = 0 remainder 16

Read bottom to top: 16, 8
16 in Base62 = 'g' (10+6)
8 in Base62 = '8'

Result: 1000₁₀ = "g8" in Base62
```

#### Issues with Base62

| Issue | Solution |
|-------|----------|
| Need unique ID first | Use ID generators (Zookeeper, Snowflake) |
| Length varies | Use padding for shorter results |

## 4. Unique ID Generation in Distributed Systems

### The Core Problem
- 10 million URLs/day = massive scale
- Single DB server won't work
- Need distributed system
- Must avoid duplicate IDs across servers

### Approach 1: Single Database Auto-Increment

```
ID | short_url | long_url
1  | ...       | ...
2  | ...       | ...
```

**Problems:**
| Issue | Impact |
|-------|--------|
| Single point of failure | System down if DB fails |
| Limited capacity | Can't handle 10M URLs/day |
| Not scalable | Only one DB server |

❌ **Not Recommended**

---

### Approach 2: Ticket Server

**Concept:** Centralized auto-increment service for all distributed apps

```
App Server 1 ─┐
App Server 2 ─┼──→ Ticket Server (Centralized ID Generator)
App Server 3 ─┘
```

**Problems:**
| Issue | Impact |
|-------|--------|
| Single point of failure | All apps affected if ticket server fails |
| Not scalable | Centralized bottleneck |

❌ **Not Recommended**

---

### Approach 3: Twitter Snowflake

**Concept:** Timestamp-based ID generation

#### Snowflake ID Structure (64 bits)
```
| 1 bit | 41 bits   | 10 bits    | 12 bits        |
|-------|-----------|------------|----------------|
| Sign  | Timestamp | Machine ID | Sequence Number|
```

| Component | Bits | Purpose |
|-----------|------|---------|
| Sign bit | 1 | Reserved (always 0) |
| Timestamp | 41 | Milliseconds since epoch |
| Machine ID | 10 | Unique identifier for each machine |
| Sequence | 12 | Counter for same millisecond |

**How Uniqueness is Guaranteed:**
- Same timestamp + Same machine → Different sequence numbers
- Same timestamp + Different machine → Different machine IDs
- Different timestamp → Already unique

✅ **Good Solution** - Used by Twitter

---

### Approach 4: Apache Zookeeper ✅ (Most Preferred)

**Definition:** A distributed application coordination service that allows distributed applications to coordinate with each other reliably.

#### How Zookeeper Works for URL Shortener

**Step 1: Divide ID Range**
```
Total Range: 0 to 3.5 trillion

Range 1: 0 - 1 million
Range 2: 1 million - 2 million
Range 3: 2 million - 3 million
...
Range N: 11 million - 12 million
```

**Step 2: Assign Ranges to Workers**
```
┌─────────────────────────────────────────┐
│              ZOOKEEPER                   │
│  ┌─────────┬─────────┬─────────┐        │
│  │Range 1  │Range 2  │Range 3  │ ...    │
│  │0-1M     │1M-2M    │2M-3M    │        │
│  └────┬────┴────┬────┴────┬────┘        │
└───────┼─────────┼─────────┼─────────────┘
        ↓         ↓         ↓
   Worker 1   Worker 2   Worker 3
   (Server1)  (Server2)  (Server3)
```

**Step 3: Range Exhaustion Handling**
- When a worker exhausts its range
- Zookeeper assigns next available unused range
- Example: Worker 1 finishes Range 1 → Gets Range 4

#### Zookeeper Advantages

| Advantage | Explanation |
|-----------|-------------|
| No synchronization needed | Each worker has exclusive range |
| Guaranteed unique IDs | No overlap between ranges |
| Scalable | Add more workers easily |
| Fault tolerant | Range reassignment on failure |

#### Trade-off
- Some ranges may go unused (if server dies mid-range)
- Acceptable because: 3.5 trillion >> 365 billion needed
- Wasting few million IDs is negligible

✅ **Most Preferred for URL Shortener**

## 5. Handling Variable Length

### Problem
Base62 encoding produces variable length outputs:
```
16 (decimal) → "g" (1 character)
1000 (decimal) → "g8" (2 characters)
```
But we need exactly **7 characters**.

### Solution: Padding

Add padding characters to reach 7 characters:
```
"g" → "g======" or "000000g"
"g8" → "g8=====" or "00000g8"
```

### Can Length Exceed 7?

**No!** Because:
- Maximum 7-character Base62 value = "ZZZZZZZ"
- "ZZZZZZZ" in decimal = ~3.5 trillion
- We only generate IDs from 0 to 3.5 trillion
- Therefore, output is always ≤ 7 characters

## 6. Complete System Design

### Architecture Diagram

```
                                    ┌─────────────┐
                                    │  Zookeeper  │
                                    │ (ID Ranges) │
                                    └──────┬──────┘
                                           │
┌──────┐     ┌───────────────┐      ┌──────┴──────┐
│      │     │               │      │             │
│ User ├────→│ Load Balancer ├─────→│ TinyURL     │
│      │     │               │      │ Server 1    │
└──────┘     └───────┬───────┘      ├─────────────┤
                     │              │ TinyURL     │
                     │              │ Server 2    │
                     │              ├─────────────┤
                     │              │ TinyURL     │
                     │              │ Server 3    │
                     │              └──────┬──────┘
                     │                     │
              ┌──────┴──────┐       ┌──────┴──────┐
              │ Data Center │       │             │
              │     1       │       │   Cache     │
              └─────────────┘       │  (Redis)    │
                                    └──────┬──────┘
                                           │
                                    ┌──────┴──────┐
                                    │  Database   │
                                    │ (Relational)│
                                    └─────────────┘
```

### Database Schema

```sql
CREATE TABLE url_mapping (
    id BIGINT PRIMARY KEY,
    short_url VARCHAR(7) UNIQUE,
    long_url TEXT NOT NULL,
    created_at TIMESTAMP
);
```

### Request Flow

#### Create Short URL (POST)
```
1. User sends long URL to Load Balancer
2. Load Balancer routes to TinyURL Server
3. Server requests unique ID from Zookeeper
4. Server converts ID to Base62
5. Server pads to 7 characters if needed
6. Server stores mapping in Database
7. Server returns short URL to user
```

#### Redirect (GET)
```
1. User requests short URL
2. Load Balancer routes to TinyURL Server
3. Server checks Cache for mapping
4. If cache miss → Query Database
5. Server returns 301/302 redirect to long URL
```

## 8. Key Interview Points

### Must-Know Calculations
```
Characters needed = 7 (for 365 billion URLs over 100 years)
Character set = 62 (0-9, a-z, A-Z)
Total combinations = 62^7 = 3.5 trillion
```

### Why Not Hash Functions?
- MD5 produces 32 characters
- SHA-1 produces 40 characters
- Trimming to 7 characters causes collisions

### Why Base62?
- Produces readable URLs
- Guaranteed unique with proper ID generation
- Predictable length with padding

### Why Zookeeper?
- Solves distributed unique ID problem
- Range-based allocation prevents conflicts
- Handles server failures gracefully
