# SQL vs NoSQL - Complete Notes

## Overview
Choosing between SQL and NoSQL is a critical decision in system design interviews. You must not only state your choice but also **justify WHY** you're choosing a particular database type.

---

## Comparison Framework

Both SQL and NoSQL are compared across **4 dimensions**:

| Dimension | What it Covers |
|-----------|----------------|
| **Structure** | How data is organized |
| **Nature** | How data is distributed |
| **Scalability** | How the system grows |
| **Property** | Guarantees provided (ACID vs BASE) |

---

**Scaling Options:**
| Type | How | SQL Support |
|------|-----|-------------|
| **Vertical** | Increase RAM, CPU, Storage | вң… Well supported |
| **Horizontal** | Add more servers (Sharding) | вҡ пёҸ Possible but complex |

**Sharding in SQL:**
- Possible but NOT intuitive
- Requires significant effort
- Breaks relational model

### 4. Property: ACID

| Property | Meaning | Guarantee |
|----------|---------|-----------|
| **A**tomicity | All or nothing transactions | Complete success or complete rollback |
| **C**onsistency | Data always valid | Rules/constraints always satisfied |
| **I**solation | Transactions don't interfere | Concurrent transactions appear sequential |
| **D**urability | Data persists | Committed data survives failures |

**Core Purpose of ACID:**
- вң… Data Integrity
- вң… Data Consistency
- вң… Reliable Transactions

## NoSQL (Not Only SQL)

### Definition
Non-relational databases designed for unstructured/semi-structured data

### 1. Structure: 4 Types

#### Type 1: Key-Value Store

```
в”Ңв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”җ

в”Ӯ  "101"  в”Ӯ  {"name": "John"...}  в”Ӯ
в”Ӯ  "102"  в”Ӯ  "Simple String"      в”Ӯ

в””в”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”ҙв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”ҳ
```

**Characteristics:**
- Simplest NoSQL type
- Value is **opaque** (can be anything: string, JSON, integer)
- Can only query by **KEY**, not by value content
- Very FAST lookups

**Example:** Amazon DynamoDB, Redis

**Use Case:** Caching, Session storage, Shopping cart

#### Soft State
- Data can change without explicit user action
- Nodes sync with each other automatically
- Example: Vector clock updates during synchronization

#### Eventual Consistency
```
Time T1: Write "John" to Node 1
         Node 1: "John" вң“
         Node 2: (old data)
         Node 3: (old data)
         
Time T2: Sync happens
         Node 1: "John" вң“
         Node 2: "John" вң“
         Node 3: "John" вң“
         
Query at T1 вҶ’ Might get stale data
Query at T2 вҶ’ Get latest data
```

## ACID vs BASE Comparison

| Aspect | ACID (SQL) | BASE (NoSQL) |
|--------|------------|--------------|
| Consistency | Immediate | Eventual |
| Availability | Lower | Higher |
| Data Integrity | Guaranteed | Best effort |
| Transactions | Strong | Weak/None |
| Use Case | Banking, Finance | Social Media, IoT |

## When to Use SQL vs NoSQL

### Use SQL When:

#### 1. Flexible Query Requirements
```
Today: SELECT * FROM users JOIN orders ON...
Tomorrow: SELECT * FROM users JOIN orders JOIN products JOIN...

SQL handles evolving, complex queries
```

#### 2. Relational/Hierarchical Data
```
Parent вҶ’ Child вҶ’ Grandchild relationships
Employee вҶ’ Department вҶ’ Company

Strong relationships between entities
```

#### 3. Data Integrity is Critical
```
Financial transactions
Banking systems
Inventory management

Cannot afford to lose or corrupt data
```

#### 4. ACID Compliance Required
```
Transaction must be:
- All or nothing
- Always consistent
- Never show partial state
```

---

### Use NoSQL When:

#### 1. Basic/Known Query Patterns
```
Always querying by: user_id, session_id, product_id
No complex joins needed
Query patterns known in advance
```

#### 2. Non-Relational Data
```
Independent documents
Key-value pairs
No complex hierarchies
```

#### 3. High Availability Required
```
System must NEVER go down
Some stale data is acceptable
24/7 availability critical
```

#### 4. High Performance at Scale
```
Millions/Billions of records
Fast read/write operations
Distributed workload
```

#### 5. Can Tolerate Inconsistency
```
Social media posts (slight delay OK)
Product catalog (not real-time critical)
Session data (can regenerate)
```

## Decision Matrix

| Factor | Choose SQL | Choose NoSQL |
|--------|------------|--------------|
| Query Complexity | Complex joins, flexible | Simple, known patterns |
| Data Structure | Relational, hierarchical | Flat, independent |
| Consistency | Must have strong | Eventual OK |
| Scale | Moderate, vertical OK | Massive, horizontal needed |
| Availability | Can tolerate downtime | Must be always up |
| Schema | Fixed, well-defined | Dynamic, evolving |
| Transactions | Critical | Not important |

## Interview Tips

1. **Never say "we can use anything"** - Always justify your choice

2. **State the trade-offs** - Show you understand both options

3. **Connect to requirements** - Link DB choice to system needs

4. **Example justification:**
   > "For this social media feed, I'll use NoSQL (Document store) because:
   > - We need high availability (users expect 24/7 access)
   > - Data is non-relational (posts are independent)
   > - We can tolerate eventual consistency (slight delay in feed OK)
   > - We need horizontal scaling for millions of users"
